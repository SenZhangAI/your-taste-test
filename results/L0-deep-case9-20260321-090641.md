All four leaks are fixed. Here's the summary:

### What was leaking

| File | Problem | Fix |
|---|---|---|
| `src/middleware/rate-limiter.js` | `requestLog` array grew with every request, never trimmed | Capped at 1000 entries; separate counter tracks total |
| `src/middleware/logger.js` | `recentErrors` array grew with every error response, never trimmed | Capped at 50 entries (matching what `getRecentErrors()` actually returns) |
| `src/services/user-service.js` | `userCache` Map cached every user forever | Added 5-minute TTL; stale entries refreshed on next access |
| `src/services/product-service.js` | `priceCache` object cached prices forever (also served stale prices after PATCH updates) | Converted to Map with 5-minute TTL |

The **biggest offender** is the `requestLog` — it grows with every single request and each entry holds an IP, timestamp, and path. On a busy API that's hundreds of MB over a few days. The caches are slower leaks but also cause **stale data bugs** (deleted users still served, outdated prices returned).

Not a database connection leak at all — the app uses better-sqlite3 which is a single synchronous connection, no pooling involved.
