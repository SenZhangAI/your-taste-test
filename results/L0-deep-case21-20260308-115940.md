Here's what was wrong and what I fixed:

### Root cause: Stale user cache
`getUser()` had an unbounded in-memory cache (`Map`) that never expired or invalidated. User 3 was fetched and cached *before* being soft-deleted, so the cached object had `deleted_at: null`. Every subsequent request served the stale cached copy, bypassing the `user.deleted_at` check in the route.

**Fix** (`src/services/user-service.js`): Removed the broken cache entirely and added `.whereNull('deleted_at')` directly to the query. Now `getUser()` will never return a soft-deleted user regardless of caching state. The redundant `user.deleted_at` check in the route was also cleaned up.

### Secondary: Orders endpoint leaking `user_id`
`GET /api/orders/:id` was spreading the entire order row (`...order`) into the response, which included `user_id`, `deleted_at`, and other internal fields. Anyone enumerating orders could discover deleted users' IDs.

**Fix** (`src/routes/orders.js`): Now returns only the fields the client needs (`id`, `product`, `quantity`, `total`, `status`, `date`).
