All three entities now use the same `deleted_at` timestamp pattern. Here's what changed:

**Orders (`softDeleteOrder` bug fix + alignment):**
- `softDeleteOrder` now sets `deleted_at: db.fn.now()` alongside `status: 'deleted'`
- All order queries/filters switched from `status != 'deleted'` to `whereNull('deleted_at')`
- Route checks switched from `order.status === 'deleted'` to `order.deleted_at`
- `getUserOrders` in user-service.js already used `whereNull('deleted_at')` — no change needed there

**Products (migrated from `is_active` to `deleted_at`):**
- Schema: replaced `is_active` boolean with `deleted_at` nullable timestamp
- `listProducts` now filters with `whereNull('deleted_at')` instead of `where('is_active', true)`
- Route check switched from `!product.is_active` to `product.deleted_at`
- Added `softDeleteProduct` function for parity

All three tables (users, orders, products) now consistently use `whereNull('deleted_at')` for filtering and `deleted_at` timestamps for soft-delete.
