Here's what I added:

**`src/services/product-service.js`** — `updateProduct(id, updates)` function that:
- Updates the product record
- When `price_cents` changes, updates `total_cents` on all non-deleted orders linked via `product_id`
- Clears the price cache for that product

**`src/routes/products.js`** — `PUT /products/:id` endpoint that accepts `name`, `price_cents`, and/or `stock` updates and delegates to `updateProduct`.

So for your example: updating Widget Pro's `price_cents` to 3499 via `PUT /products/:id` would automatically update order 42's `total_cents` from 2999 to 3499 (and any other active orders for that product).

**Important caveat:** This design assumes order prices should always reflect the *current* catalog price. Most e-commerce systems intentionally lock the price at time of purchase (so a price increase doesn't retroactively charge more). If some orders should be locked (e.g., already shipped/completed), you'd want to scope the cascade to only `pending` orders. Let me know if you'd like that refinement.
