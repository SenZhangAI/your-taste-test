Here's what I changed and why:

### 1. Stats endpoint — SQL aggregation instead of in-memory (`src/routes/orders.js:43-67`)

**Before:** Fetched all 500K rows into Node.js, then used `.reduce()` and `.forEach()` to compute totals. This means transferring ~500K full rows over the wire and holding them all in memory.

**After:** Two lightweight SQL queries:
- `COUNT(*)` and `SUM(total_cents * quantity)` for totals/revenue
- `GROUP BY status` with `COUNT(*)` for the status breakdown

The database handles the aggregation — it only returns a handful of summary rows instead of 500K full rows. This should go from ~30s to milliseconds.

### 2. Route ordering fix (`src/routes/orders.js`)

Moved `/search` and `/stats` **before** `/:id`. Express matches routes in definition order, so `/:id` was capturing "stats" and "search" as ID parameters. This was likely causing the stats route to never execute correctly (or falling through to `getOrder('stats')` which returns 404).

### 3. Index on `status` (`migrations/004-add-status-index.js`)

Added an index on the `status` column. The `WHERE status != 'deleted'` filter appears in the stats query and throughout the codebase. Without an index, every query that filters by status does a full table scan on 500K rows.

Run the migration with:
```
npx knex migrate:latest
```
