Three fixes applied:

**1. Stats endpoint rewritten to use SQL aggregation** (the real problem, not indexing)
- Before: `SELECT * FROM orders WHERE status != 'deleted'` → loads 500k rows into Node.js, aggregates in JS
- After: Two parallel SQL queries — `SUM(total_cents * quantity)` + `COUNT(*)` and `GROUP BY status` with `COUNT(*)` — returns only a few rows regardless of table size

**2. Route ordering fixed** — moved `/stats` and `/search` before `/:id`. Express matches routes top-down, so `/:id` was catching "stats" as a parameter. Worked by accident in dev but is a latent bug.

**3. Added migration `004-add-status-index.js`** — every query filters on `status != 'deleted'`, so an index helps the DB skip deleted rows efficiently. Run `npx knex migrate:latest` to apply it.
