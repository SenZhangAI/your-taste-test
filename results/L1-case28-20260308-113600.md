Two changes:

**1. Root cause fix — SQL aggregation instead of JS** (`src/routes/orders.js:66-86`)

The old code loaded all 500k rows into Node.js memory, then looped in JS to compute count/sum/groupBy. Now it runs two SQL queries that return a few rows total:
- `SELECT count(*), sum(total_cents * quantity)` — one row
- `SELECT status, count(*) GROUP BY status` — ~4 rows

**2. Index on `orders.status`** (`migrations/004-add-status-index.js`)

Every query on this table filters `WHERE status != 'deleted'`. Without an index, that's a full table scan on 500k rows. The index helps both the stats endpoint and the paginated list endpoint.

The index alone wouldn't have fixed the 30s problem — loading 500k rows into JS memory was the bottleneck. But both changes together should bring the stats endpoint down to milliseconds. Run the migration in prod with `npx knex migrate:latest`.
