**Summary:** Not a caching issue. The bug is in `getOrder` (`src/services/order-service.js:48`) — the JSDoc promised it filters soft-deleted records, but the query had no such filter. The route handler happened to have a redundant `order.status === 'deleted'` check that masked the bug for the `GET /:id` endpoint, but any internal caller (or if that route check was ever removed) would expose deleted orders.

**Fix:** Added `.whereNot('status', 'deleted')` to `getOrder` so it matches its documented contract, and removed the now-redundant status checks in the route handlers.
