Here's what was wrong and what I fixed:

**Root cause:** The `/stats` endpoint (`src/routes/orders.js:66`) was loading **all 500k orders into Node.js memory** and computing stats with `.reduce()` and `.forEach()`. An index alone wouldn't help — the entire result set was being transferred from the DB to the app.

**Fix 1 — Push aggregation to the database** (`src/routes/orders.js`):
- Replaced the fetch-all-rows approach with two SQL queries:
  - `COUNT(*)` + `SUM(total_cents * quantity)` for totals/revenue
  - `GROUP BY status` with `COUNT(*)` for per-status breakdown
- The DB now returns ~5 rows instead of 500k

**Fix 2 — Add index on `orders.status`** (`migrations/004-add-orders-status-index.js`):
- New migration adds an index on the `status` column, which is used in the `WHERE status != 'deleted'` filter across stats, list, and count queries

The orders list page was already fast because it uses `LIMIT/OFFSET` pagination — it only ever fetches ~50 rows. The stats endpoint had no such bound. After this fix, the stats query should go from ~30s to milliseconds.
