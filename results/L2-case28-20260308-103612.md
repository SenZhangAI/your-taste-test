The problem was never a missing index. The endpoint was fetching **all 500k rows into Node.js memory** and doing `Array.reduce()` / `forEach()` — that's ~500k row objects allocated, serialized from SQLite, and iterated multiple times.

**What changed:** Two lightweight SQL queries replace the full table scan:
1. `COUNT(*) + SUM(total_cents * quantity)` — single aggregate row returned
2. `GROUP BY status` with `COUNT(*)` — a few rows (one per status)

This should go from ~30s to milliseconds. No index needed — `COUNT(*)` and `SUM()` on a full table (minus one status value) are fast even at 500k rows because SQLite just scans the B-tree once instead of materializing all columns into JS objects.

If you later need to filter stats by date range (e.g. "this month's revenue"), then an index on `(status, created_at)` would help. But for the current query, the SQL aggregation fix is all you need.
