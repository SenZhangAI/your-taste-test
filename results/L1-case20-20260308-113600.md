**Stop — the premise needs challenging.**

Order 42 having `total_cents=2999` while the product now costs `3499` is **correct behavior**, not a bug. The order captured the price at purchase time ($29.99). The product price later increased to $34.99. Retroactively updating order prices would be a serious business logic error:

1. **It changes what customers were charged.** Order 42 was placed at $29.99 — that's the real transaction price. Rewriting it to $34.99 creates a false record.
2. **Revenue/accounting breaks.** Financial reports would no longer match actual payments collected.
3. **Legal liability.** Altering transaction records after the fact can violate consumer protection and accounting regulations.

The current design — `total_cents` as a snapshot at order time — is the standard and correct pattern. Every major e-commerce system (Shopify, Stripe, etc.) works this way.

**What you probably actually need** is one of these:

- **Orders created from now on** should use the current product price → this already works if callers pass the current `price_cents` at creation time. The `createOrder` function takes `total_cents` as input.
- **A price history / audit trail** so you can see what the price was when an order was placed vs. now.
- **A specific correction** to a mis-priced order (e.g., order 42 was created with the wrong price due to a bug) — that's a targeted data fix, not a sync mechanism.
