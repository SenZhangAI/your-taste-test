**Summary of what was wrong and what was fixed:**

| Issue | Location | Fix |
|-------|----------|-----|
| `getOrder` had no soft-delete filter | `order-service.js:48` | Added `where('status', '!=', 'deleted')` |
| `getUserOrders` checked `deleted_at` which was never set | `user-service.js:36` | Changed to `where('status', '!=', 'deleted')` |
| `softDeleteOrder` didn't set `deleted_at` | `order-service.js:72` | Now sets `deleted_at` alongside status |
| Route handlers had workaround checks | `routes/orders.js:32,82` | Removed redundant `status === 'deleted'` checks — service handles it now |

**Root cause:** The JSDoc was wrong — `getOrder` never filtered deleted records. The route handler partially masked this with its own check, but any other consumer of `getOrder` (including `createOrder` which returns the newly created order via `getOrder`) was exposed. `getUserOrders` had a separate bug — it filtered on `deleted_at` which `softDeleteOrder` never set, so deleted orders leaked through that path too.

Order #42 will now correctly return 404.
