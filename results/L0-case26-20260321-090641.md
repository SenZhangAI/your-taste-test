Here's what was happening:

- **`getProduct(id)`** queries the DB directly — no cache. The PATCH endpoint uses this, so the response correctly shows the new price.
- **`getProductPrice(productId)`** uses an in-memory `priceCache` that's populated on first call and **never invalidated**. Order creation uses this function for price validation, so it keeps seeing the stale cached price.

The fix adds `invalidatePriceCache()` and calls it in the PATCH handler right after the DB update, so the next call to `getProductPrice()` will re-fetch from the database.
