**This is not a bug — it's correct, intentional behavior. You should not add sync logic.**

Here's what's happening:

1. **`orders.total_cents` is a price snapshot** — it captures the price at the time the order was placed. Order 42 was created when Widget Pro cost 2999 cents ($29.99). The price later went up to 3499 cents ($34.99). The order record is correct.

2. **The `product_id` FK on orders** is for associating the order with a product (e.g., for stock management, reporting), not for keeping prices in sync.

3. **`getProductPrice()` with its cache** is used during order *creation* to look up the current price — the comment says exactly this: "Used by order creation to validate price at purchase time." It's not meant for refreshing existing orders.

**Why syncing would be harmful:**
- Order 42 represents a real transaction at $29.99. Changing it to $34.99 would mean the order record no longer matches what the customer was actually charged.
- Invoices, refunds, and financial reporting all depend on `total_cents` being the historical truth.
- If you sync prices, every past order's revenue changes retroactively — accounting breaks.

**What you might actually want instead** (depending on the use case):
- **Show current vs. purchased price in admin UI** — query both `orders.total_cents` and `products.price_cents` and display them side by side, so ops can see the delta without mutating history.
- **Price new orders correctly** — the current `createOrder` flow already takes `total_cents` as input. If you want to ensure new orders use the latest catalog price, validate at creation time using `getProductPrice()`.
