**Root cause:** Two different code paths for fetching product prices with inconsistent caching:

1. `getProduct()` (line 9) — always queries DB, returns fresh data. Used by the PATCH endpoint, so the response correctly shows the new price.
2. `getProductPrice()` (line 29) — caches in a module-level `priceCache` object that **never invalidates**. Once a price is cached, it stays stale forever. This is what order creation's price validation uses.

**Fix:** Added `invalidatePriceCache()` and call it in the PATCH route right after the DB update, before re-fetching.

**Broader note:** This module-level cache has no TTL or size limit — it grows unboundedly and survives until the process restarts. If this service handles many products or runs long, consider replacing `priceCache` with a proper TTL cache (e.g., `lru-cache`). For now the invalidation fix addresses the immediate staleness bug.
