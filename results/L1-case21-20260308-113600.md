**Root cause:** `userCache` is a `Map` that never invalidates. Once user 3 was fetched before deletion, the cached copy (without `deleted_at`) was served forever. The route's `user.deleted_at` check worked correctly — it just never saw the updated DB row.

The orders endpoint was a red herring — it wasn't leaking anything. `listOrders` and `searchOrders` both filter `status != 'deleted'` and don't expose user data directly.

**What I changed:** Removed the unbounded, never-invalidating in-memory user cache entirely. `getUser()` now always reads from DB, so `deleted_at` is seen immediately.

If you need caching back for performance later, it should use TTL-based expiration or explicit invalidation on user mutations — not an eternal `Map`.
