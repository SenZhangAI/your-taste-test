Not a caching issue — the bug is in `getOrder()`. The JSDoc says it filters soft-deleted records, but the implementation doesn't:

**`src/services/order-service.js:32`** — `getOrder` queries by `id` only, with no deleted filter:
```js
return db('orders').where({ id }).first();
```

The route layer (`orders.js:30`) compensates with `order.status === 'deleted'` check, so the `GET /:id` endpoint actually works correctly. But any other caller of `getOrder()` (including `createOrder` at line 52, which returns the newly created order through it) gets no filtering.

The real question is: how is the customer seeing the deleted order? If they're hitting `GET /orders/:id`, the route-level check should catch it. Are they hitting a different endpoint, or is there another code path that calls `getOrder` directly?

That said, the fix is straightforward — `getOrder` should match its documented contract:

```js
export async function getOrder(id) {
  return db('orders').where({ id }).whereNot('status', 'deleted').first();
}
```

This moves the filter to the authoritative layer (service) instead of relying on every caller to check. The route-level `order.status === 'deleted'` check then becomes redundant but harmless.
