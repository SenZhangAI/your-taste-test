Codebase explored. Key findings:

- **Knex + SQLite**, layered architecture (routes → services → db)
- `GET /api/orders` paginates with default limit 50, filters out `status='deleted'`, supports sort/status params
- Response maps fields: `product_name` → `product`, formats price as `$X.XX`, date as `YYYY-MM-DD`
- ~5000 orders in seed data

You suggested "loop through pages" but that's unnecessary overhead — we have direct DB access. Here's the recommended approach:

**Direct DB query** — add a `getAllOrders()` service function that queries the DB without pagination, then stream rows as CSV. No reason to loop through the paginated API when we own the data layer.

Key design decisions:
1. **CSV columns**: match the existing response shape — `id, product, quantity, total, status, date`
2. **Filtering**: same as list endpoint (exclude deleted), optionally support `?status=` filter
3. **Streaming**: for 5000 rows it's not critical, but we can use Knex's `.stream()` to keep memory flat — worth it?
4. **Content-Disposition**: set header for browser download with filename like `orders-2026-03-08.csv`
