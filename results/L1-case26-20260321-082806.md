**Root cause:** `priceCache` in `product-service.js` is a write-once, never-invalidated in-memory cache. The PATCH route was also bypassing the service layer by writing directly to DB, so even adding invalidation to the service wouldn't have helped without routing the update through it.

**What changed:**

1. **`product-service.js`** — Added `updateProductPrice()` that updates DB and `delete`s the cache entry, so the next `getProductPrice()` call fetches fresh data.
2. **`routes/products.js`** — PATCH handler now calls the service function instead of hitting DB directly. Removed unused `db` import.

**Breadth-scan note:** The `priceCache` pattern itself is fragile — no TTL, no size limit, and in a multi-process deployment (cluster/PM2) each process would have its own stale copy. If this scales beyond a single process, consider replacing it with a TTL-based cache or just dropping it entirely (a single `WHERE id = ?` query is fast enough).
