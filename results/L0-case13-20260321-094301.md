Here's what changed:

1. Added `'total_cents'` to `SORTABLE_FIELDS` 
2. Added a mapping so `sort=price` resolves to the actual `total_cents` column

Since `total_cents` is an integer column, the database will sort numerically — no string comparison issues. `?sort=price` will now work as documented in `docs/API.md`.

Note: unknown sort values (anything other than `created_at`, `updated_at`, or `price`) still silently fall back to `created_at`. You may want to return a 400 error for invalid sort fields instead, but that's a separate concern.
