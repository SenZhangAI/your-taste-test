**Root cause:** `getOrder` returned deleted orders despite its JSDoc claiming otherwise. `listOrders` and `searchOrders` both correctly filter `WHERE status != 'deleted'`, but `getOrder` was missing this filter.

The route handler had a band-aid check (`order.status === 'deleted'`) that partially masked the bug, but any other caller of `getOrder` (including `createOrder` which uses it to return newly created orders) would have leaked deleted records.

**What changed:**
1. `order-service.js:48` — added `where('status', '!=', 'deleted')` to match the documented contract
2. `routes/orders.js` — removed redundant `order.status === 'deleted'` checks in GET and DELETE handlers (now handled at the service layer where it belongs)
