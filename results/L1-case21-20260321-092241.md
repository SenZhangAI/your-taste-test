Summary of what was wrong and what changed:

**Root cause**: `userCache` in `user-service.js` — an in-memory `Map` that never invalidates. Once user 3 was fetched before deletion, the stale cached copy (without `deleted_at`) was returned on every subsequent request, bypassing the `deleted_at` check in the route.

**Changes made**:

| File | Fix |
|---|---|
| `services/user-service.js` | Removed broken cache. `getUser()` now filters `deleted_at` at the query level. |
| `routes/users.js` | Removed redundant `user.deleted_at` check (now handled by query). |
| `services/order-service.js` | `listOrders` and `searchOrders` now join on `users` to exclude deleted users' orders. |
| `routes/orders.js` | `/stats` endpoint joins on `users` to exclude deleted users. `POST /` rejects orders for deleted users. |

The orders endpoint wasn't the cause of the profile bug, but you were right that it was leaking deleted user data — their orders appeared in all listing/search/stats responses.
