Before jumping in — your suggestion to "loop through pages" caught my eye. That's the wrong approach here. Let me explain why and propose a better design.

**Why not loop through the paginated endpoint:**
- The existing `GET /api/orders` formats data for JSON display (formatted prices like `$12.99`, formatted dates) — CSV should use raw values for downstream processing
- Looping through pages means N+1 queries and assembling results in memory — unnecessary complexity when you can just query all orders directly
- `getOrderTotal()` treats `total_cents` as dollars (returns `total_cents * quantity`) — this is a unit naming bug. The CSV should export raw `total_cents` and `quantity` so the consumer can compute correctly

**Recommended approach: query the DB directly, stream CSV**

- Add `GET /api/orders/export` that queries all non-deleted orders in a single query
- Stream rows as CSV (no JSON formatting, no pagination)
- Use raw column values: `total_cents` (integer), ISO dates, etc.
- Set `Content-Type: text/csv` and `Content-Disposition: attachment` headers
- No external CSV library needed — the data is simple enough to escape manually

**Columns:** `id, product_name, quantity, total_cents, status, created_at`

**Trade-offs considered:**
1. ~~Loop through paginated API~~ — extra complexity, wrong data format, unit bug propagation
2. **Single query + stream** (recommended) — simple, correct, handles large datasets via streaming
3. ~~Cursor-based batching~~ — overkill for SQLite, useful if this were Postgres with millions of rows

One thing to flag: `getOrderTotal` multiplies `total_cents * quantity`, but the column is named `total_cents` which implies it's already the line total. Worth clarifying — but for the CSV export I'd just export the raw columns and let the consumer decide.
