**Root cause:** `userCache` is a `Map` that caches users forever with no invalidation. When user 3 was fetched before deletion, the cached copy (without `deleted_at`) was stored. After soft-deleting in the DB, subsequent requests to `GET /api/users/3` still got the stale cached version, which had no `deleted_at` — so the 404 check on line 22 of `users.js` never triggered.

The orders endpoint is innocent here — `getUserOrders` already queries the DB directly with a `whereNull('deleted_at')` filter and has no caching.
