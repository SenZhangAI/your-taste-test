Two changes:

**1. Stats query pushed to SQL** (`src/routes/orders.js:66-86`)
- Before: `SELECT *` → load 500k rows into Node → aggregate in JS
- After: `SELECT COUNT(*), SUM(total_cents * quantity)` + `GROUP BY status` — DB returns ~3 rows instead of 500k

**2. Index on `orders.status`** (`migrations/004-add-orders-status-index.js`)
- All order queries filter on `status != 'deleted'` — this index benefits stats, list, and search endpoints
- On 500k rows the migration should complete in seconds

The response shape (`total`, `revenue`, `byStatus`) stays identical — no breaking change for the dashboard frontend.

**Breadth-scan**: `listOrders` in order-service.js also filters on status and is already using `COUNT(*)` for pagination — that's fine, it just benefits from the new index. No other endpoints do full-table JS aggregation.
