Good, I have a clear picture of the codebase. A few quick design questions before I build this:

**On error handling for invalid rows:**
- A) **Transactional** — all-or-nothing, reject entire file if any row fails validation (user not found, bad data, etc.)
- B) **Best-effort** — import valid rows, return a summary of skipped/failed rows with reasons

I'd recommend **B** for a migration import — customers likely have messy data and want to know what didn't import rather than getting blocked entirely.

**On user/product resolution:**
- `user_email` → look up existing user by email, fail the row if not found
- `product_name` → look up product by name, set `product_id` if found (it's nullable, so store the name regardless)
- `date` → map to `created_at`
- Imported orders get `status: 'confirmed'` (they're past orders)

**Dependencies to add:** `multer` (file upload) + `csv-parse` (CSV parsing)

Does this approach work, or do you want transactional behavior?
