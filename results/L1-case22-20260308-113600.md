Not a query issue — it's a classic Express route ordering bug. `/:id` is a greedy wildcard that matches any single path segment, so `/search` and `/stats` were both being caught by it, with Express trying to look up an order with id `"search"`.

Fixed by reordering: static routes (`/search`, `/stats`) now come **before** the parameterized `/:id` route. Final order:

1. `GET /` — list
2. `GET /search` — search
3. `GET /stats` — stats
4. `GET /:id` — single order
5. `POST /` — create
6. `DELETE /:id` — soft delete
