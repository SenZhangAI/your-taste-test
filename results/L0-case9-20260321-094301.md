Found **5 memory leaks** — and the database connections aren't the main culprit. Here's the breakdown:

### Critical: Unbounded arrays that grow forever

1. **`src/middleware/logger.js:6,24-26`** — `recentErrors` array. Every 400+ response pushes to it, but entries are never removed. The `getRecentErrors()` function slices the last 50 for display, but the underlying array grows forever.

2. **`src/middleware/rate-limiter.js:4,29`** — `requestLog` array. Every single request is pushed here for "dashboard monitoring" but nothing ever cleans it up. The rate-limit `hits` Map gets cleaned, but `requestLog` doesn't. This is likely the **biggest contributor** to your memory growth.

### High: Unbounded caches with no eviction

3. **`src/services/user-service.js:3-17`** — `userCache` Map caches every user looked up, with no TTL or size limit. Over days, this accumulates every user that's ever been queried.

4. **`src/services/product-service.js:7,23-31`** — `priceCache` object does the same for product prices — no eviction, no TTL.

### Medium: Missing shutdown cleanup

5. **`src/db.js` + `src/index.js:31-35`** — No SIGTERM/SIGINT handler to close the database connection, so rolling restarts can leak handles.

Want me to fix these? The typical approach would be:
- Cap `recentErrors` to the last 50 entries (trim on insert)
- Remove `requestLog` or give it a rolling window
- Add LRU/TTL eviction to both caches
- Add a graceful shutdown handler for the DB connection
