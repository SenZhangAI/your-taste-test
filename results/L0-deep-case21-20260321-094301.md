**Root cause:** `userCache` is a `Map` with no TTL and no invalidation. Once a user object is cached (with `deleted_at: null`), setting `deleted_at` in the DB has no effect on cached responses. The `user.deleted_at` check in the route (`users.js:22`) always sees the stale cached value.

**The orders endpoint is a red herring** — `getUserOrders` already correctly filters with `.whereNull('deleted_at')`, and `listOrders`/`searchOrders` filter by `status != 'deleted'`. The leak was entirely from the user cache bypassing the DB.
