Here's what was wrong and what I fixed:

**Root cause:** The `/stats` endpoint (`src/routes/orders.js:67`) was loading all 500k order rows into Node.js memory, then computing count, sum, and grouping in JavaScript. With 500k rows that's a massive data transfer + in-memory processing on every request.

**Fix 1 — Push aggregation to the database** (`src/routes/orders.js`):
- `COUNT(*)` and `SUM(total_cents * quantity)` replace fetching all rows for totals/revenue
- `GROUP BY status` with `COUNT(*)` replaces the JS `.forEach` loop
- The database now returns ~3-4 rows instead of 500k

**Fix 2 — Add index on `orders.status`** (`migrations/004-add-status-index.js`):
- Every query in this app filters on `WHERE status != 'deleted'` — the list endpoint, search, stats, count — but there was no index on `status`
- This was masked in dev with small data but causes full table scans at 500k rows

After deploying, run `npx knex migrate:latest` to apply the new index migration.
