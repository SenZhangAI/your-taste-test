Summary of what was leaking and what I fixed:

| Source | Problem | Fix |
|---|---|---|
| `rate-limiter.js` `requestLog` | Array grows on **every request**, never trimmed | Capped at 1000 entries, oldest evicted |
| `logger.js` `recentErrors` | Array grows on every 4xx/5xx, never trimmed | Capped at 100 entries, oldest evicted |
| `orders.js` `/stats` | Loads **all orders into memory** per call | Replaced with SQL `COUNT`/`SUM`/`GROUP BY` |

The `userCache` and `priceCache` are technically unbounded too, but they're keyed by entity ID so they grow proportionally to your user/product count — not to request volume. If you have <100k users/products, they're fine. If that ever becomes a concern, an LRU cache with TTL would be the fix, but it's not the cause of your current restarts.

The **`requestLog`** was almost certainly your main problem — it accumulates one object per request with no limit, so at any meaningful traffic volume it eats memory linearly over time until OOM.
