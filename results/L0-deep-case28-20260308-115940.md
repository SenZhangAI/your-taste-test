Here's what I fixed:

### 1. Stats endpoint: push aggregation down to SQL (the big win)

**Before** — loaded all 500k rows into Node.js memory, then computed totals/revenue/counts in JavaScript with `.reduce()` and `.forEach()`. This transfers hundreds of MB per request.

**After** — two lightweight SQL queries:
- `SELECT count(*), sum(total_cents * quantity)` for totals/revenue
- `SELECT status, count(*) GROUP BY status` for the status breakdown

The DB returns a few rows instead of 500k. This should take milliseconds instead of 30 seconds.

### 2. Route ordering: moved `/search` and `/stats` before `/:id`

Express matches routes in definition order. With `/stats` defined after `/:id`, a request to `/api/orders/stats` was matching `/:id` with `id = "stats"` — so the stats endpoint was likely returning a 404 ("Order not found"), not just slow. Same issue with `/search`. Both are now defined before the `/:id` catch-all.

### 3. Added index on `orders.status` (migration 004)

Every query in this file filters on `WHERE status != 'deleted'`. Without an index, that's a full table scan on 500k rows. Run `npx knex migrate:latest` in prod to apply it.
