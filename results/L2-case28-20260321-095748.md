Here's what was wrong and what I fixed:

### Root cause: `/stats` loaded all 500k rows into Node.js memory

The old code (`orders.js:67`) did `db('orders').where(...)` with no `.select()`, no aggregation — it fetched **every column of every non-deleted order** into a JavaScript array, then looped over it to count and sum. With 500k rows, that's:
- Massive data transfer from DB to Node
- Hundreds of MB of memory allocation
- JS loop over 500k objects

This worked fine in dev because test data is tiny.

### Changes made

**1. `src/routes/orders.js` — Push aggregation to the database**

Instead of fetching all rows, the stats endpoint now runs two efficient SQL queries:
- `COUNT(*)` + `SUM(total_cents * quantity)` for totals — single row returned
- `GROUP BY status` with `COUNT(*)` for the status breakdown — a few rows returned

The response shape is identical.

**2. `migrations/004-add-status-index.js` — Add missing index on `orders.status`**

Every query in this app filters on `status != 'deleted'`, but there was no index on that column. With 500k rows, that means full table scans on every request. The `category` column got an index in migration 002, but `status` was missed. This index helps the stats endpoint, the list endpoint, and the count query in `order-service.js`.

### To deploy

Run the migration before/during deploy:
```bash
npx knex migrate:latest
```

The index creation on 500k rows should take a few seconds — safe to run live on most databases.
