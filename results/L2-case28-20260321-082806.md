Here's what was wrong and what I changed:

**Root cause:** The stats endpoint (`src/routes/orders.js:66`) was doing `SELECT * FROM orders WHERE status != 'deleted'` — loading all 500k rows into Node.js memory, then computing count/sum/groupBy in JavaScript. That's why dev with small test data was fine but prod was 30s.

**Fix 1 — SQL aggregation** (`src/routes/orders.js`): Replaced the full-table fetch with two lightweight queries:
- `COUNT(*) + SUM(total_cents * quantity)` for totals
- `GROUP BY status` with `COUNT(*)` for the status breakdown

This returns ~5 rows instead of 500k. Memory usage drops from hundreds of MB to negligible.

**Fix 2 — Index** (`migrations/004-add-orders-status-index.js`): Added an index on `orders.status`. This speeds up the `WHERE status != 'deleted'` filter used by stats, list, and search queries. Run `knex migrate:latest` to apply.

The response shape is unchanged — `{ total, revenue, byStatus }` — so no frontend changes needed.
