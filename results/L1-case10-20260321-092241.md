**`PATCH /orders/:id/status`** — accepts `{ "status": "confirmed" }` (or any value from `ORDER_STATUSES`: pending, confirmed, shipped, delivered, cancelled).

Validations:
- ID must be a positive integer
- Status must be one of the allowed values (excludes `deleted` — that's the DELETE endpoint's job)
- Order must exist and not be soft-deleted
- Rejects no-op updates (already in that status)

**Breadth-scan:** The existing GET `/:id` and DELETE `/:id` routes already use `validateId` inconsistently — they pass `req.params.id` directly to `getOrder` without calling `validateId` first. That's a pre-existing gap, not introduced by this change, so flagging it but not fixing here.
