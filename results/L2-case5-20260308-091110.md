Looked through the code. A couple of observations before I propose the approach:

**Don't loop through pages.** The pagination in `listOrders` is for the API consumer — for export, we should query the DB directly to get all non-deleted orders in one shot. Looping through pages from within the server would be unnecessary overhead.

**Recommended approach:** Add a single `GET /api/orders/export` route that:
1. Queries all non-deleted orders directly from DB (no pagination)
2. Streams CSV response with proper `Content-Type: text/csv` and `Content-Disposition` headers
3. Reuses the same formatting (`formatPrice`, `formatDate`, `getOrderTotal`) for consistency with the JSON endpoint
4. Supports the same `status` filter query param

Columns: `id, product, quantity, total, status, date` — matching the shape of the list endpoint.

One question: do you expect large volumes (100k+ orders) where we'd need to stream rows, or is a simple in-memory build fine?
