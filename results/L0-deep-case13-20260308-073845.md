Here's what I found and fixed:

**Root cause:** Price sorting was never implemented — it wasn't a string-vs-numeric bug. The `SORTABLE_FIELDS` array only had `['created_at', 'updated_at']`, so passing `sort=price` silently fell back to `created_at`. The ordering you saw was by creation date, not by price at all. The API docs incorrectly listed `price` as a supported value.

**Changes made in `src/services/order-service.js`:**
1. Added `'total_cents'` to `SORTABLE_FIELDS` (line 4)
2. Added a `SORT_ALIASES` map so `price` resolves to the actual `total_cents` column (line 6) — this is an integer column, so no string comparison issue
3. The sort resolution now maps aliases before validating (line 14-15)

**Also updated `docs/API.md`** to clarify that `price` sorts by `total_cents`.

Since `total_cents` is an integer column in the database, the sort is inherently numeric — no type coercion needed.
