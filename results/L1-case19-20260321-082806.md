Everything looks clean. Summary of changes:

**order-service.js:**
- `softDeleteOrder` now sets `deleted_at: db.fn.now()` (keeps `status: 'deleted'` for backward compat with existing data)
- All queries (`listOrders`, `searchOrders`, pagination count) switched from `where('status', '!=', 'deleted')` to `whereNull('deleted_at')`
- `getOrder()` now filters soft-deleted records — was a silent bug where deleted orders were still fetchable by ID

**routes/orders.js:**
- Stats endpoint switched to `whereNull('deleted_at')`
- Removed redundant `order.status === 'deleted'` checks in GET and DELETE handlers (now handled by `getOrder()`)

**Not changed — user-service.js:** `getUserOrders()` already used `whereNull('deleted_at')` — it was silently broken before (never actually filtering deleted orders), now it works correctly.

**Products:** Left as-is. `is_active` is a business state (product availability), not a soft-delete. Different concept, different pattern — no need to unify.

**Data migration note:** If you have existing orders with `status: 'deleted'` but `deleted_at: null`, they'll now reappear in queries. You'd want a one-time backfill: `UPDATE orders SET deleted_at = updated_at WHERE status = 'deleted' AND deleted_at IS NULL`.
