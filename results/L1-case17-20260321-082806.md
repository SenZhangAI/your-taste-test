**`PATCH /api/orders/:id`** — updates order fields before shipment.

Key behaviors:
- **Allowed fields**: `product_name`, `quantity`, `total_cents`, `product_id`, `status`
- **Guard**: Only `pending` / `confirmed` orders can be modified. Returns `409` if the order is already `shipped`, `delivered`, or `cancelled`
- **Validation**: ID must be valid integer, `quantity` and `total_cents` must be positive integers, `status` must be from the known statuses list (can't set `deleted` via PATCH — that's the DELETE endpoint's job)
- **404** for missing/deleted orders, **400** for bad input, **409** for status conflicts

**Breadth-scan**: The existing GET `/:id` and DELETE `/:id` routes don't use `validateId` — they pass raw `req.params.id` to `getOrder`. This is a pre-existing gap (knex will just return no results for non-numeric IDs, so it's not a bug, but inconsistent). I used `validateId` in the new endpoint to follow the validators pattern. Worth noting but not worth changing the existing endpoints unprompted.
