**What was wrong**: The stats endpoint fetched all 500k order rows into Node.js memory, then did `Array.reduce` / `Array.forEach` to compute totals. With 500k rows that means:
- ~200MB+ transferred from DB to app
- Full table scan (no index on `status`)
- All computation in JS instead of the database

**What changed**:

1. **`src/routes/orders.js`** — Replaced the fetch-all-then-aggregate with two SQL queries:
   - `COUNT(*) + SUM(total_cents * quantity)` for totals/revenue
   - `GROUP BY status` with `COUNT(*)` for the status breakdown
   - Database returns 2-5 rows instead of 500k

2. **`migrations/004-add-orders-status-index.js`** — Adds index on `orders.status`, which benefits the stats endpoint, `listOrders`, and `searchOrders` (all filter on `status != 'deleted'`).

After deploying, run `npx knex migrate:latest` to apply the index. The stats query should go from ~30s to milliseconds.
