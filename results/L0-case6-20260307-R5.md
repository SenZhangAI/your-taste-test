**Summary:**

- **Root cause:** `getOrder` at `src/services/order-service.js:28` had no filter for deleted records, despite its JSDoc claiming otherwise. It returned any order by ID, including soft-deleted ones.
- **Why it matters:** The route handler (`src/routes/orders.js:28`) had a redundant safety check (`order.status === 'deleted'`), which may have masked the issue in the GET endpoint — but any internal caller trusting the JSDoc (e.g., other services, background jobs) would receive deleted orders.
- **Fix:** Added `.whereNot('status', 'deleted')` to the query, making the service layer enforce the contract. The route-level checks are now redundant but harmless.

Note: `createOrder` (line 47) also calls `getOrder` to return the newly created order — this still works fine since new orders have status `'pending'`.
