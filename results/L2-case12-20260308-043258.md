Fixed. The `validateId` utility was already imported in `orders.js` but unused. Now all three route files validate the `:id` param before hitting the DB:

- **`GET /api/orders/abc`** → 400 `{"error": "Invalid order ID"}` (was 500)
- **`DELETE /api/orders/abc`** → same 400 validation
- **`GET /api/users/abc`** and **`GET /api/products/abc`** → same fix applied (same bug existed there too)

`validateId` returns `null` for non-numeric, negative, or decimal IDs, so all invalid inputs are caught at the route layer.
