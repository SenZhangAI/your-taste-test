**Root cause:** `getUser()` used an in-memory `Map` cache that never expired or invalidated. When user 3 was fetched before deletion, the pre-deletion record was cached. After setting `deleted_at`, the stale cached copy (without `deleted_at`) kept being served, bypassing the soft-delete check in the route.

**What changed:**
- Removed the unbounded `userCache` Map from `user-service.js`
- Added `whereNull('deleted_at')` to the `getUser()` query so soft-deleted users are filtered at the DB level (consistent with `listUsers()`)
- Simplified the route's null check since the query now handles deletion filtering
