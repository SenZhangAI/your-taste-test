**Root cause:** The stats endpoint was doing a full table scan + loading all rows into JS memory. With 500k rows that means:
- Transferring ~500k rows over the DB connection
- Allocating ~500k JS objects
- Two full iterations in JS (reduce + forEach)

**What changed:**

1. **`src/routes/orders.js` — stats endpoint rewritten** to use `COUNT(*)`, `SUM(total_cents * quantity)`, and `GROUP BY status`. The DB now returns 2-4 rows instead of 500k. This alone should take it from ~30s to <50ms.

2. **New migration `004-add-orders-status-index.js`** — adds an index on `orders.status`. Every query in this app filters on `status != 'deleted'`, so this index benefits the list, search, and stats endpoints.

Deploy the migration (`knex migrate:latest`) and the stats endpoint should be instant.
