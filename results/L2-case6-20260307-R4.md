**Summary:**

- **Root cause:** `getOrder()` at `order-service.js:28` queried by `id` only, with no deleted filter — despite the JSDoc claiming otherwise. Any direct `GET /api/orders/:id` request could return soft-deleted orders.
- **Fix:** Added `.whereNot('status', 'deleted')` to the query in `getOrder`, consistent with how `listOrders` already filters on line 11.
- **Cleanup:** Removed the now-redundant `order.status === 'deleted'` checks in both route handlers (`GET /:id` and `DELETE /:id`), since `getOrder` now returns `null` for deleted orders — the filtering responsibility belongs in the service layer, not scattered across callers.
