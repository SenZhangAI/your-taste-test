**Summary of the fix:**

The customer was seeing deleted order #42 through `GET /api/users/:id` — not `GET /api/orders/42`.

**Root cause:** Soft-delete mechanism mismatch. `softDeleteOrder()` sets `status: 'deleted'`, but `getUserOrders()` was filtering on `whereNull('deleted_at')` — a column that's never populated. So deleted orders always passed through.

**Secondary issue:** `getOrder()` JSDoc claimed it filtered deleted records but didn't. The route handler had a compensating manual check, which masked the bug at the order endpoint but left the user endpoint exposed.

**Changes:**
- `order-service.js`: `getOrder()` now filters `status != 'deleted'` at the query level (matches its contract)
- `user-service.js`: `getUserOrders()` now filters on `status != 'deleted'` instead of `whereNull('deleted_at')`
- `routes/orders.js`: Removed redundant `order.status === 'deleted'` checks since `getOrder()` handles it now
