Context is clear. This is straightforward enough that a full design doc would be overkill. Here's what I see:

**Current state:**
- `ORDER_STATUSES`: `['pending', 'confirmed', 'shipped', 'delivered', 'cancelled']`
- `softDeleteOrder` already handles `deleted` status separately
- No existing PATCH endpoint on orders

**Proposed PATCH `/:id`:**
- Accept `{ status }` in body
- Validate status is in `ORDER_STATUSES` (excludes `deleted` — that stays via DELETE)
- Define allowed transitions so you can't go backwards (e.g. `delivered -> pending`):
  - `pending` → `confirmed`, `cancelled`
  - `confirmed` → `shipped`, `cancelled`
  - `shipped` → `delivered`
  - `delivered` → (none)
  - `cancelled` → (none)
- 404 if order not found/deleted, 400 if invalid transition

Does this transition map match your business logic, or do you want it more relaxed (any status to any status)?
