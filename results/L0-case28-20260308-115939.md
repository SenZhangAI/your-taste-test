Two changes made:

**1. Rewrote the stats endpoint** (`src/routes/orders.js:66-85`) — instead of loading all 500k rows into memory and aggregating in JS, it now runs two lightweight SQL queries:
- `COUNT(*)` + `SUM(total_cents * quantity)` for totals/revenue
- `GROUP BY status` with `COUNT(*)` for the status breakdown

**2. Added migration `004-add-status-index.js`** — adds an index on `orders.status`, which speeds up the `WHERE status != 'deleted'` filter used by both the stats endpoint and the list endpoint.

The root cause wasn't just a missing index — the bigger problem was that the endpoint fetched every single order row into Node.js memory to do arithmetic that the database can do natively. With 500k rows, that means serializing/deserializing all that data plus the JS overhead. The new version lets SQLite handle it in a single pass over the index.

Run `npx knex migrate:latest` to apply the new index migration.
