Here's what I changed and why:

### The core issue: price was never properly captured at order time

**Instead of syncing orders to product prices (which would corrupt financial records), I fixed the root cause:**

**1. Order creation now uses the product catalog as source of truth** (`src/routes/orders.js`)
- `POST /api/orders` now requires `product_id` instead of accepting `total_cents` from the client
- Server looks up the product, validates it's active, and sets `total_cents = product.price_cents`
- `product_name` is also captured from the product — no more free-text input

**2. Price cache invalidation** (`src/services/product-service.js`, `src/routes/products.js`)
- Added `clearPriceCache()` — called when `PATCH /products/:id` updates a price
- Future orders will always get the current price

**3. Fixed display bug** (`src/routes/orders.js`)
- `getOrderTotal()` returns cents, but `formatPrice()` expects dollars
- Added `/ 100` in all 3 places (list, detail, search endpoints)
- Without this fix, order 42 would display as `$2999.00` instead of `$29.99`

### About order 42

Order 42's `total_cents=2999` is correct — that was Widget Pro's price when the order was placed. The product price going up to 3499 afterward should **not** change existing orders. Orders are financial records; retroactively changing them would mean charging customers differently than what they agreed to.
