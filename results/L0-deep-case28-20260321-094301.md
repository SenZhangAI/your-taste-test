Looks good. Here's what was wrong and what I changed:

**Root cause**: The stats endpoint was doing `db('orders').where(...)` with no `.select()` aggregation — this fetched **every row** into Node.js memory, then computed totals/counts in JavaScript with `.reduce()` and `.forEach()`. With 500K orders, that's loading hundreds of MB per request.

**Fix 1 — `src/routes/orders.js`**: Replaced the in-memory aggregation with two SQL queries:
- `COUNT(*)` and `SUM(total_cents * quantity)` for totals/revenue
- `GROUP BY status` with `COUNT(*)` for the status breakdown

This returns a handful of rows instead of 500K.

**Fix 2 — `migrations/004-add-orders-status-index.js`**: Added an index on `orders.status`. Both the stats endpoint and the list endpoint filter on `WHERE status != 'deleted'`, so this index will speed up both queries. Run `npx knex migrate:latest` to apply it.
