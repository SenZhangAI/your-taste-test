The fix:

1. Added `total_cents` to `SORTABLE_FIELDS` — this is the actual numeric column in the DB, so sorting is numeric (not string)
2. Added a `SORT_ALIASES` map so `sort=price` resolves to `total_cents`

The root cause wasn't a string-vs-numeric comparison bug — price sorting simply didn't exist. The `sort=price` parameter was silently falling back to `created_at`, making results appear in creation order rather than by price. The API docs claimed it was supported, but the code never implemented it.
